# Indy Nav MVP — Design Spec

## Overview

A web app providing unofficial advice for navigating Indy Pass mountains. The MVP focuses on a single feature: a sortable table of Northeast Indy Pass mountains ordered by closing date for the 2026 season, helping users plan spring skiing.

## Goals

- Show all Northeast Indy Pass mountains with their 2026 closing dates
- Sort by closing date (latest first) so users can plan spring trips
- List already-closed mountains below open mountains
- List unknown mountains in between
- Filter by "as of" date to help plan for a specific upcoming weekend
- Indicate confidence level for each closing date (some mountains announce firm dates, others say "conditions permitting")
- Link to official mountain websites for further info

## Non-Goals (for MVP)

- Map view
- Travel time calculations
- Favorites / follows (future — will use localStorage)
- User accounts or auth
- Other regions beyond Northeast
- Indy Pass vs Indy+ distinction
- Production deployment (runs on localhost)

## Tech Stack

- **Framework:** Next.js (App Router)
- **Styling:** Tailwind CSS + shadcn/ui with IBM Carbon-inspired theme
- **AI:** Vercel AI SDK + Google Gemini Flash-Lite (data pipeline only, not runtime)
- **Language:** TypeScript
- **Deployment:** localhost (possibly homelab later)

## Architecture

```
indy-nav/
├── src/app/                    # Next.js app router
│   ├── page.tsx                # Home — mountain closing dates table
│   ├── layout.tsx              # Root layout
│   └── globals.css             # Global styles
├── src/components/             # React components
│   └── ui/                     # shadcn/ui components
├── src/lib/                    # Utilities, types, constants
│   └── types.ts                # TypeScript interfaces
├── scripts/                    # Data pipeline
│   └── fetch-closing-dates.ts  # Gemini-powered closing date extraction
├── data/                       # Static + generated data
│   ├── mountains.json          # Hand-maintained seed data
│   └── closing-dates.json      # Generated by pipeline
└── tailwind.config.ts          # Carbon-inspired theme tokens
```

Two distinct subsystems:

1. **Data pipeline** (`scripts/`) — Standalone script run manually. Fetches mountain web pages, sends content to Gemini Flash-Lite to extract closing dates, writes structured JSON.
2. **Web app** (`src/`) — Next.js app that reads the JSON file and renders a sortable table.

No database, no auth, no runtime AI calls. The JSON file is the "database."

## Data Model

### Seed Data (`data/mountains.json`)

Hand-maintained list of Northeast Indy Pass mountains:

```typescript
interface MountainSeed {
  id: string;          // slug, e.g. "magic-mountain"
  name: string;        // "Magic Mountain"
  state: string;       // "VT"
  region: string;      // "Northeast"
  websiteUrl: string;  // Official mountain website
  closingDateUrl: string; // URL most likely to contain closing date info
}
```

### Enriched Data (`data/closing-dates.json`)

Generated by the pipeline:

```typescript
interface Mountain {
  id: string;
  name: string;
  region: string;
  state: string;
  closingDate: string | null;       // "2026-04-12" or null if unknown
  closingDateSource: string;        // URL where the date was found
  closingDateConfidence: "high" | "medium" | "low";
  lastUpdated: string;              // ISO timestamp of last pipeline run
  websiteUrl: string;
}
```

`closingDateConfidence` captures Gemini's certainty:
- **high** — Firm date announced (e.g., "Season ends April 12")
- **medium** — Tentative or conditional (e.g., "Expected to close mid-April")
- **low** — Inferred or ambiguous (e.g., "Conditions permitting")

## Data Pipeline

Script: `scripts/fetch-closing-dates.ts`

### Process

1. Read `data/mountains.json` (seed list)
2. For each mountain, fetch the `closingDateUrl` page content
3. Send page content to Gemini Flash-Lite via Vercel AI SDK's `generateObject()` with a Zod schema
4. Prompt: "Extract the closing date for the 2025-2026 ski season from this page. Return the date in YYYY-MM-DD format, your confidence level (high/medium/low), and a brief explanation of your confidence assessment."
5. Write enriched results to `data/closing-dates.json`
6. Log summary: mountains processed, dates found, confidence breakdown, failures

### Behavior

- **Sequential processing** with a small delay between requests to avoid rate limits
- **Incremental by default** — Skip mountains with `lastUpdated` within the last 24 hours. Override with `--force` flag.
- **Resilient** — If a fetch or AI call fails for one mountain, log the error and continue to the next
- **Run command:** `npx tsx scripts/fetch-closing-dates.ts [--force]`
- **API key:** Requires `GOOGLE_GENERATIVE_AI_API_KEY` environment variable (via `.env.local`, not committed)

## UI

### Page: Mountain Closing Dates (home)

A single-page app with a sortable data table.

#### Table Columns

| Column | Description |
|--------|-------------|
| Mountain | Name, links to official website |
| State | Two-letter state code |
| Closing Date | Formatted date (e.g., "Apr 12") |
| Confidence | Badge: green (high), yellow (medium), red (low) |
| Last Updated | Relative time (e.g., "2 hours ago") |

#### Default Sort

Closing date descending (latest closers at top). Mountains with unknown dates grouped at the bottom.

#### Styling — Carbon-Inspired Theme

- Neutral gray palette (IBM Carbon gray scale)
- Clean, minimal borders
- Dense but readable row spacing
- 14px body type
- Minimal decoration — data-forward aesthetic

### No MVP Features

- No search or filters
- No responsive/mobile optimization (nice to have, not required)
- No dark mode

## Future Features (not in scope)

These inform the architecture but are not built in MVP:

- Travel time calculations from user's location
- Favorites / follows (localStorage)
- Additional regions
- Links to OpenSnow reports and official mountain reports
- Season date ranges (opening + closing)
